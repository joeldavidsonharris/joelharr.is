JobMatcha - brewing a better job hunt

The background

JobMatcha is my first attempt at a full-stack website. It came about in January 2025 when I was moving to Australia. I quit my job in New Zealand and was looking for something in Australia.

I found the whole process needlessly manual. The issue was that I knew what I wanted, but I couldn’t effectively filter jobs based on that. A lot of the important information was only available when reading the whole job advert.

I liken it to Tinder – imagine if, instead of Tinder matches, you were provided with random profiles and had to search each one to see if you were really a match. You probably wouldn't use it for very long.

Searching for job matches in JobMatcha

I wanted job matches that came to me, and so, JobMatcha was born. You input your experience and what you’re looking for; it scrapes Seek, reads the job ads, and matches against your input.

A job match. You can click the seek logo to apply

And overview of the architecture

I got a domain name (jobmatcha.org), hosted it on Digital Ocean (for about five dollars a month), and hooked it up to GitHub so that it would automatically update whenever I publish code changes.

The learnings

Ultimately, this was a learning exercise. I wanted to go through the motions of making a product, to give myself a baseline understanding and to feel some of the pain that motivates a better approach.

The good

  • Learned how to get domains, get hosting, and deploy
  • Learned more JavaScript, CSS, and HTML
  • Learned some Flask and API stuff
  • Used a lot of generative AI-enhanced workflows for the front-end code
  • Nice colours, nice branding

The bad

  • Building on top of an unsupported API is not very smart – every time Seek updates their front-end my service breaks
  • I found it too taxing for the time saved
    • I needed to essentially rewrite my CV to save looking at around 20 job listings on a single platform
    • Most of the “advanced search” use cases are already covered by Seek’s existing search system
  • It has no clear path to monetisation
  • It has high costs that scale with usage
    • Want to avoid large usage-based costs until I am happy committing capital
  • Designing a product as I’m building it is hard

For next time

  • Spend a bit of time doing directed research and design prior to building
    • Things to understand:
      • Market size
      • Competition
      • Ability to scale
      • Monetisation
      • How to reach users
      • Is there a network effect involved?
    • Things to design:
      • User journey
      • Wireframes
  • Use a front-end framework such as Next.js
    • This can hopefully speed up UI development time once the learning curve is over
  • Build monetisation into the next product
    • Need practice integrating Stripe
